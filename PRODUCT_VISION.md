# PRODUCT_VISION.md — PaperPilot 产品愿景

## 1. 一句话定位

PaperPilot 是一个面向高校科研场景的论文阅读、证据回溯、多论文对比与研究辅助 Agent 平台。

它当前是一个可运行的论文 RAG + 多工具 Agent 系统，未来希望逐步演进成一个真正能帮助硕士生、博士生、科研人员和高校课题组提高文献阅读与研究整理效率的 AI 工具。

---

## 2. 为什么要做这个项目

高校科研场景里，很多痛点不是“缺一个聊天机器人”，而是：

1. 论文太多，读不完。
2. 多篇论文方法相似，容易混淆。
3. 大模型回答看似流畅，但不知道依据来自哪里。
4. 做综述、开题、组会、论文写作前，需要反复整理文献脉络。
5. 传统文献管理工具偏存储和引用管理，缺少面向内容理解和证据追踪的 AI 工作流。
6. 普通 ChatGPT / Claude 上传论文后可以问答，但缺少本地知识库、可控检索、证据展示、Agent Trace 和持续迭代能力。

PaperPilot 要解决的不是“让 AI 替人科研”，而是让 AI 成为科研阅读和知识整理过程中的可靠助手。

---

## 3. 目标用户

### 3.1 近期目标用户

- 硕士研究生
- 博士研究生
- 正在写论文 / 开题 / 综述的学生
- 需要快速理解论文的 AI 工程师
- 需要管理小型论文库的课题组成员

### 3.2 中长期目标用户

- 高校课题组
- 科研助理
- 企业技术调研团队
- AI 应用工程团队
- 需要围绕大量技术文档做知识问答和证据追踪的团队

---

## 4. 核心场景

### 4.1 单篇论文理解

用户上传一篇论文后，可以问：

- 这篇论文解决了什么问题？
- 主要贡献是什么？
- 方法流程是什么？
- 实验设置是什么？
- 主要结论是什么？
- 局限性是什么？
- 哪些段落可以支持这个结论？

系统回答时必须返回证据来源，而不是只给总结。

---

### 4.2 多篇论文对比

用户上传多篇同方向论文后，可以问：

- Paper A 和 Paper B 方法有什么区别？
- 这几篇论文分别解决了什么问题？
- 它们的数据集、指标和实验结论有什么不同？
- 哪篇论文更适合作为我课题的 baseline？
- 这几篇论文的发展脉络是什么？

这是 PaperPilot 未来最重要的差异化方向之一。

普通 RAG 容易在多篇相似论文中发生 source-level confusion，因此后续需要重点做 source-aware retrieval、metadata filter、hybrid retrieval 和 evaluation。

---

### 4.3 证据回溯

PaperPilot 不应该只回答“是什么”，还要回答：

- 这个结论来自哪篇论文？
- 来自哪个 chunk？
- 检索距离是多少？
- rerank 后排第几？
- 系统为什么认为证据充分或不充分？
- 如果证据不足，系统为什么拒答？

科研场景中，可信比流畅更重要。

---

### 4.4 论文笔记生成

未来可以支持自动生成结构化论文笔记：

- 研究问题
- 研究动机
- 核心方法
- 主要贡献
- 实验设置
- 关键结果
- 局限性
- 可引用证据
- 与其他论文的关系

但所有生成内容都应尽量关联到 evidence source。

---

### 4.5 Related Work 辅助

未来可以支持从多篇论文中生成综述素材，例如：

- 按方法分类
- 按时间线整理
- 按研究问题整理
- 按数据集和任务整理
- 输出带来源的 related work 草稿素材

注意：系统不应定位为“自动代写论文”，而应定位为“带证据的科研材料整理助手”。

---

## 5. 产品原则

### 5.1 Evidence First

所有关键回答都应优先基于检索到的论文证据。

如果没有足够证据，应明确拒答或提示当前知识库不足。

不要为了显得智能而编造答案。

---

### 5.2 Traceable AI

系统不仅要返回 answer，还要返回：

- Retrieved Context
- source
- distance
- retrieval_rank
- rerank result
- context_sufficient
- context_metrics
- tool_used
- route_reason
- workflow_path
- fallback_used

可观测性是 PaperPilot 的核心竞争力之一。

---

### 5.3 Research Workflow, Not Just Chat

PaperPilot 不应该停留在“聊天框问答”。

长期应围绕科研工作流设计：

- 上传论文
- 管理论文库
- 生成论文笔记
- 多论文对比
- 证据回溯
- 生成综述素材
- 导出 Markdown / Word / Excel
- 支持组会和开题准备

---

### 5.4 Local-first and Privacy-aware

高校和科研场景中，论文、课题方向、实验方案可能比较敏感。

短期可以使用本地 PDF + 远程 LLM API。

中长期要考虑：

- 本地知识库
- 私有化模型
- 本地 embedding
- 可替换 LLM provider
- 用户数据隔离
- 不上传敏感内容的模式

---

### 5.5 Incremental Refactoring

不要一次性把项目改成复杂 SaaS。

每次只做一个清晰的小改动：

- 保持系统可运行
- 保持 /ask 可用
- 保持 /reload_kb 可用
- 保持 Agent Trace 可用
- 保持 Retrieved Context 可用
- 改完后补测试或至少补 smoke test

---

## 6. 阶段路线

### Stage 0：面试级 AI 应用工程项目

目标：

- 项目能稳定演示
- README 清楚
- 架构清楚
- 能讲 RAG、Agent、FastAPI、Django、Trace
- 能支撑 AI 应用工程 / RAG / Agent 面试

当前项目已经基本处在这个阶段。

---

### Stage 1：作品级科研 RAG Agent

目标：

- 检索更准
- 多论文 source confusion 明显降低
- 有轻量 eval 闭环
- Agent Trace 更好看
- 页面更像真实产品
- 代码结构更适合持续迭代

优先任务：

1. source-aware retrieval
2. eval_questions.json + eval 脚本
3. Agent Trace 页面优化
4. embedding 缓存
5. FastAPI lifespan 与状态管理优化

---

### Stage 2：真实科研助手原型

目标：

- 找 3–5 个研究生或科研用户试用
- 观察真实问题
- 记录 bad case
- 做论文笔记生成
- 做多论文对比表
- 支持 Markdown / Word 导出

这一阶段开始验证真实科研痛点，而不是只靠自己想象功能。

---

### Stage 3：高校科研 AI 产品雏形

目标：

- 支持用户项目空间
- 支持多论文库管理
- 支持课题组级文献库
- 支持更好的权限和数据隔离
- 支持更稳定的部署
- 支持私有化模型或本地模型选项

这一阶段才考虑商业化和创业验证。

---

## 7. 当前最重要的产品判断

PaperPilot 暂时不要追求“大而全”。

短期最该打磨的是：

1. 多论文检索准确性
2. 证据可信度
3. Agent Trace 可解释性
4. 小规模 eval 评估闭环
5. 产品演示体验

不要过早做：

- 多租户 SaaS
- 复杂权限系统
- 复杂支付系统
- 复杂前端框架重写
- 多 Agent 大架构
- 和科研主流程无关的炫技功能

---

## 8. 给 AI Coding Agent 的产品提醒

当你作为 coding agent 参与 PaperPilot 开发时，请始终记住：

1. 这个项目不是普通聊天机器人。
2. 这个项目的核心价值是科研论文场景下的证据可信、内容回溯和多论文理解。
3. 不要为了增加功能破坏现有 RAG / Agent / Trace 闭环。
4. 不要把项目重构成无法运行的“大架构草图”。
5. 每次改动前先说明它解决了哪个科研痛点或工程痛点。
6. 每次改动后说明如何验证。
7. 优先做可演示、可评估、可持续迭代的小步改进。
